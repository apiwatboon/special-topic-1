01 GitHub คืออะไร

GitHub คือบริการบนเว็บ (web-based platform) สำหรับ เก็บและจัดการซอร์สโค้ด โดยทำงานอยู่บนระบบควบคุมเวอร์ชัน (Version Control System) ที่ชื่อ Git พูดง่าย ๆ คือเป็น "พื้นที่เก็บโค้ดออนไลน์" ที่ช่วยให้เรา:

  • เก็บโค้ดไว้บนคลาวด์ ไม่หายแม้คอมพัง
  • ย้อนดูประวัติการแก้ไขทุกครั้งได้ (ใครแก้ แก้อะไร เมื่อไหร่)
  • ทำงานเป็นทีมพร้อมกันหลายคนโดยไม่ทับงานกัน
  • แชร์ผลงานให้คนอื่นเห็น นำไปใช้ หรือช่วยพัฒนาต่อได้
เปรียบเทียบให้เห็นภาพ: ถ้า Google Drive คือที่เก็บไฟล์เอกสารทั่วไป GitHub ก็คือ "Google Drive สำหรับโค้ด" ที่ฉลาดกว่ามาก — มันจำได้ว่าไฟล์เปลี่ยนไปอย่างไรในแต่ละบรรทัด และให้หลายคนแก้ไฟล์เดียวกันแล้วรวมงานเข้าด้วยกันได้

02 ประวัติความเป็นมา (มาจากไหน)

2005
กำเนิด Git — Linus Torvalds (ผู้สร้าง Linux) พัฒนา Git ขึ้นมา เพื่อใช้จัดการซอร์สโค้ดของ Linux kernel ที่มีนักพัฒนาร่วมกันจำนวนมหาศาล

อ้างอิง: ประวัติ Git (Wikipedia)

2008
เปิดตัว GitHub — ก่อตั้งโดยทีม 4 คน เพื่อทำให้การใช้ Git ง่ายขึ้นผ่านหน้าเว็บ (สามคนแรกคือผู้ใช้หมายเลข 1-3 ของ GitHub เลย)

อ้างอิง: ประวัติ GitHub (Wikipedia)

2018
Microsoft ซื้อ GitHub ด้วยมูลค่าราว 7.5 พันล้านดอลลาร์ และยังคงเปิดให้ใช้งานได้ฟรีสำหรับนักพัฒนาทั่วไป

อ้างอิง: ประกาศทางการของ Microsoft

2021–ปัจจุบัน
ยุค AI — เปิดตัว GitHub Copilot (2022) ผู้ช่วยเขียนโค้ดด้วย AI และกลายเป็นแพลตฟอร์มศูนย์กลางของนักพัฒนากว่า 100 ล้านคนทั่วโลก

อ้างอิง: เปิดตัว Copilot · Dohmke อำลา GitHub (2025)

03 Git กับ GitHub ต่างกันอย่างไร

นี่คือจุดที่หลายคนสับสนบ่อยที่สุด — สองอย่างนี้ ไม่ใช่สิ่งเดียวกัน แต่ทำงานคู่กัน

Git
จดบันทึกประวัติโค้ด
เครื่องของเรา Local · ใช้ออฟไลน์ได้
git push →
← git pull
GitHub
คลาวด์ออนไลน์ แชร์ · ทำงานเป็นทีม

Git อยู่ในเครื่องเราคอยจดบันทึกการเปลี่ยนแปลง แล้วใช้คำสั่ง push (ส่งขึ้น) และ pull (ดึงลง) รับ-ส่งกับ GitHub บนคลาวด์ — GitHub ตั้งอยู่บน Git แต่เป็นคนละสิ่งกัน

Git

โปรแกรม / เครื่องมือ

  • ระบบควบคุมเวอร์ชันที่ติดตั้งในเครื่องเรา
  • ทำงานผ่านคำสั่ง (command line)
  • ใช้งานได้แม้ไม่มีอินเทอร์เน็ต
  • เป็นซอฟต์แวร์โอเพนซอร์ส ฟรี

GitHub

บริการออนไลน์

  • เว็บไซต์ที่ใช้เก็บ repository ของ Git บนคลาวด์
  • มีหน้าจอกราฟิกใช้งานง่าย
  • ต้องเชื่อมต่ออินเทอร์เน็ต
  • เพิ่มฟีเจอร์ทำงานเป็นทีม (Issues, Pull Request ฯลฯ)
จำง่าย ๆ: Git คือ "เครื่องยนต์" ที่ทำงานในเครื่องเรา ส่วน GitHub คือ "โรงจอดรถออนไลน์" ที่เก็บผลงานและให้คนอื่นมาดูได้ — ทางเลือกอื่นที่คล้าย GitHub ก็มี เช่น GitLab และ Bitbucket

05 แนวคิดและคำศัพท์พื้นฐาน

ก่อนเริ่มใช้งานจริง มาทำความรู้จักคำศัพท์ที่จะเจอบ่อย

คลิกที่การ์ดคำศัพท์เพื่อไฮไลท์และเล่นภาพเคลื่อนไหวของคำนั้น

Repository (Repo)

clone / pull / fork ต่างกันอย่างไร?

สามคำนี้ชวนสับสนที่สุด เพราะดู "ดึง/ก๊อป" เหมือนกัน — แต่ต่างกันที่ "จากไหนไปไหน" และ "ใช้ตอนไหน"

คำสั่งก๊อปจากไหน → ไปไหนใช้ตอนไหน
CloneGitHub → เครื่องเรา (ทั้ง repo, ครั้งแรก)เริ่มทำงานกับ repo ครั้งแรก ดึงมาทั้งก้อน
PullGitHub → เครื่องเรา (เฉพาะที่อัปเดตใหม่)repo ที่ clone ไว้แล้ว อยากดึงงานล่าสุดของทีมมา
Forkrepo คนอื่น (GitHub) → บัญชีเรา (GitHub)อยากแก้ต่อ repo ที่เราไม่ได้เป็นเจ้าของ
จำง่าย ๆ: clone กับ pull = ดึงจาก GitHub มาที่ เครื่องเรา (clone ครั้งแรกทั้งก้อน · pull ดึงเฉพาะของใหม่ที่เพิ่มเข้ามา) ส่วน fork = ก๊อป repo คนอื่นไปไว้ใน บัญชี GitHub ของเรา (ยังอยู่บนคลาวด์ ไม่ได้ลงเครื่อง)

06 การสมัครสมาชิก (ทีละขั้น)

  1. เข้าเว็บไซต์ ไปที่ github.com แล้วกดปุ่ม Sign up
  2. กรอกอีเมล ใช้อีเมลที่ใช้งานจริง (แนะนำอีเมลมหาวิทยาลัยเพื่อขอสิทธิ์นักศึกษาภายหลัง)
  3. ตั้งรหัสผ่าน ควรเป็นรหัสที่คาดเดายาก มีตัวอักษร ตัวเลข และอักขระพิเศษ
  4. ตั้งชื่อผู้ใช้ (username) ชื่อนี้จะปรากฏใน URL ของคุณ เช่น github.com/your-name — เลือกให้เป็นทางการ เพราะใช้ในพอร์ตสมัครงานได้
  5. ยืนยันตัวตน ทำแบบทดสอบยืนยันว่าไม่ใช่บอท แล้วกด Create account
  6. ยืนยันอีเมล เปิดอีเมลที่ลงทะเบียน แล้วกดลิงก์ยืนยัน — เสร็จเรียบร้อย!
เคล็ดลับนักศึกษา: สมัคร GitHub Student Developer Pack ด้วยอีเมลมหาวิทยาลัย จะได้ใช้เครื่องมือพรีเมียมหลายอย่าง ฟรี เช่น GitHub Copilot, โดเมนเว็บ และเครดิตคลาวด์ต่าง ๆ

07 ทำความรู้จักเมนูต่าง ๆ บน GitHub

หลังล็อกอิน หน้าแรกของ GitHub จะมีแถบเมนูทางด้านซ้าย แต่ละอันมีหน้าที่ดังนี้

Homeหน้าฟีดรวม แสดงความเคลื่อนไหวจากโปรเจกต์และผู้คนที่เราติดตาม รวมถึงทางลัดไป repo ที่ใช้บ่อย
Issuesรวม "ปัญหา/งานที่ต้องทำ" ทั้งหมดจากทุก repo ที่เกี่ยวข้องกับเรา (ที่เราสร้างไว้ หรือถูกมอบหมาย)
Pull requestsรวมคำขอรวมโค้ด (PR) ทั้งหมดที่เราเปิดไว้ หรือถูกขอให้ช่วยรีวิวก่อนรวมเข้างานหลัก
Projectsกระดานจัดการงานแบบ Kanban สำหรับวางแผนและติดตามสถานะงานของทีม
Discussionsกระดานพูดคุย ถาม-ตอบ และแลกเปลี่ยนความคิดเห็นของชุมชนในแต่ละโปรเจกต์
Codespacesสภาพแวดล้อมเขียนโค้ดบนคลาวด์ (เหมือนเปิด VS Code ในเบราว์เซอร์) เขียนและรันโค้ดได้โดยไม่ต้องติดตั้งอะไรในเครื่อง
Copilotผู้ช่วย AI ของ GitHub ถาม-ตอบ ให้ช่วยเขียน อธิบาย หรือแก้โค้ดให้ได้
Exploreสำรวจค้นหาโปรเจกต์น่าสนใจ โปรเจกต์มาแรง (trending) และหัวข้อต่าง ๆ บน GitHub
Marketplaceแหล่งรวมแอปและเครื่องมือเสริม (เช่น GitHub Actions) ที่นำมาต่อกับ repo เพื่อเพิ่มความสามารถ
MCP registryทะเบียนรวม MCP server (Model Context Protocol) ซึ่งเป็นมาตรฐานเปิดสำหรับให้ AI/เอเจนต์เชื่อมต่อกับเครื่องมือและข้อมูลภายนอก — ใช้ค้นหาและติดตั้ง MCP server มาใช้งาน

08 อย่าสับสน: Repository ≠ Projects

หลายคนสับสนระหว่าง Repository กับ Projects เพราะคำว่า "โปรเจกต์" ใช้ได้หลายความหมาย — แต่สองอย่างนี้คนละหน้าที่กัน

Repository

ที่เก็บโปรเจกต์จริง

  • เก็บโค้ด ไฟล์ และประวัติการแก้ไขทั้งหมด
  • เป็น "ที่ที่โค้ดอยู่" จริง ๆ
  • เปรียบเหมือนกล่องเก็บของ / โฟลเดอร์โปรเจกต์
  • โดยทั่วไป 1 งาน = 1 repo

Projects

เครื่องมือจัดการงาน

  • กระดานวางแผน/ติดตามงานแบบ Kanban
  • บอกว่างานไหนยังไม่ทำ / กำลังทำ / เสร็จแล้ว
  • ไม่เก็บโค้ด
  • 1 Project ดึงงานจากหลาย repo มารวมบนกระดานเดียวได้
สรุปสั้น ๆ: ถ้าพูดลอย ๆ ว่า "โปรเจกต์ของฉัน" มักหมายถึง repo นั่นแหละ — แต่ "GitHub Projects" (เมนูในเว็บ) คือกระดานจัดการงาน เป็นคนละสิ่งกับที่เก็บโค้ด

10 การอัปโหลดโปรเจกต์ขึ้น GitHub

มี 2 วิธีหลัก เลือกแบบที่ถนัด — ลองกดสลับแท็บดูได้เลย

  1. กดปุ่ม + มุมขวาบน → เลือก New repository
  2. ตั้งชื่อ repo เช่น my-first-project
  3. เลือก Public (ใครก็เห็น) หรือ Private (เฉพาะเรา)
  4. ติ๊ก Add a README file แล้วกด Create repository
  5. ในหน้า repo กด Add file → Upload files
  6. ลากไฟล์ทั้งหมดมาวาง แล้วกด Commit changes — เสร็จ!
เหมาะกับผู้เริ่มต้น หรือไฟล์จำนวนไม่มาก ไม่ต้องติดตั้งอะไรเพิ่ม

ขั้นแรก ติดตั้ง Git ในเครื่องก่อน จากนั้นเปิด Terminal / Git Bash:

# 1) ตั้งค่าชื่อและอีเมล (ทำครั้งแรกครั้งเดียว)
git config --global user.name "ชื่อของคุณ"
git config --global user.email "อีเมลของคุณ"

# 2) เข้าไปในโฟลเดอร์โปรเจกต์ แล้วเริ่มต้น Git
cd my-first-project
git init

# 3) เพิ่มไฟล์ทั้งหมด และบันทึก (commit)
git add .
git commit -m "first commit"

# 4) เชื่อมกับ repo บน GitHub (คัดลอก URL จากหน้า repo)
git branch -M main
git remote add origin https://github.com/USERNAME/REPO.git

# 5) ส่งขึ้น GitHub
git push -u origin main
วิธีนี้เหมาะกับการทำงานจริง อัปเดตโค้ดบ่อย ๆ และทำงานเป็นทีม

13 ลงมือทำจริง: เขียนโปรแกรมแล้วขึ้น GitHub

มาลองทำครบวงจรแบบ step by step — เขียนโปรแกรมง่าย ๆ เอาขึ้น GitHub แล้วลองแก้โค้ดและ commit ใหม่ เพื่อดูว่า commit ทำงานอย่างไรและผลเป็นอย่างไร

เลือกภาษา:

ขั้นที่ 1 · เขียนโปรแกรม

สร้างไฟล์ชื่อ hello.c เขียนโค้ดง่าย ๆ:

#include <stdio.h>

int main() {
    printf("Hello, GitHub!\n");
    return 0;
}

สร้างไฟล์ชื่อ hello.py เขียนโค้ดง่าย ๆ:

print("Hello, GitHub!")

ขั้นที่ 2 · เอาขึ้น GitHub ครั้งแรก

  1. สร้าง repository ใหม่ (เช่น hello-chello-py) แบบ Public ติ๊ก Add a README file
  2. ในหน้า repo กด Add file → Upload files
  3. ลากไฟล์โปรแกรมของคุณ (hello.chello.py) เข้ามาวาง
  4. ใส่ข้อความ commit เช่น first commit: add hello แล้วกด Commit changes
ตอนนี้ไฟล์ขึ้นไปอยู่บน GitHub แล้ว และเกิด commit แรก ขึ้นมา 1 ครั้ง

ขั้นที่ 3 · แก้โค้ด แล้ว commit ใหม่

  1. กดที่ไฟล์โปรแกรมในหน้า repo
  2. กดไอคอน ดินสอ (Edit) มุมขวาบนของไฟล์
  3. แก้โค้ด เช่น เพิ่มข้อความต้อนรับอีกบรรทัด:
#include <stdio.h>
int main() {
printf("Hello, GitHub!\n");
+ printf("ยินดีต้อนรับสู่ Git!\n");
return 0;
}
print("Hello, GitHub!")
+print("ยินดีต้อนรับสู่ Git!")
  1. เลื่อนลงล่างไปที่ช่อง Commit changes
  2. ใส่ข้อความ commit เช่น เพิ่มข้อความต้อนรับบรรทัดที่สอง
  3. กด Commit changes

ผลลัพธ์: ประวัติ commit

กดที่ Commits ในหน้า repo จะเห็นประวัติเรียงจากใหม่ไปเก่า — ตอนนี้มี 2 commit:

เพิ่มข้อความต้อนรับบรรทัดที่สองคุณ committed ไม่กี่วินาทีที่แล้ว · a1b2c3d
first commit: add helloคุณ committed สักครู่ก่อน · 9f8e7d6
  • แต่ละ commit = สแน็ปช็อตของโค้ด ณ จุดนั้น พร้อมข้อความอธิบายว่าทำอะไร
  • กดที่ commit ใด ๆ จะเห็น diff ว่าบรรทัดไหนถูกเพิ่ม (สีเขียว) หรือลบ (สีแดง)
  • ถ้าแก้แล้วพัง ย้อนกลับไปเวอร์ชันของ commit ก่อนหน้าได้เสมอ

ทำขั้นที่ 3 แบบคำสั่ง Git (ทางเลือก)

ถ้าทำงานในเครื่องด้วย Git โดยตรง วงจรแก้ → commit → push เป็นแบบนี้ (เหมือนกันทั้ง C และ Python):

# หลังแก้ไฟล์ในเครื่องแล้ว
git add .
git commit -m "เพิ่มข้อความต้อนรับบรรทัดที่สอง"
git push
เคล็ดลับ: ข้อความ commit ที่ดีควรสั้น กระชับ และบอกว่า "ทำอะไร" เพื่อให้ย้อนดูประวัติแล้วเข้าใจง่าย

14 commit & push

รวมทุกสถานการณ์ที่นักศึกษาจะเจอจริง — ทำตามทีละขั้นได้เลย ไม่ต้องเดา

กรณี A · เริ่มโปรเจกต์ใหม่จากศูนย์ (ในเครื่อง → ขึ้น GitHub)

  1. สร้าง repo เปล่าบน GitHub ก่อน — กด + มุมขวาบน → New repository → ตั้งชื่อจริง เช่น my-project → ยังไม่ต้องติ๊ก Add README/gitignore → Create repository
  2. คัดลอก URL จริงจากปุ่มเขียว Code → HTTPS (เช่น https://github.com/USERNAME/my-project.git)
  3. เปิด Git Bash แล้วเข้าโฟลเดอร์โปรเจกต์ในเครื่อง:
cd C:/myproject
  1. ต้องมีไฟล์อย่างน้อย 1 ไฟล์ในโฟลเดอร์ก่อน (เช่น index.html หรือ README.md) เพราะ Git ไม่เก็บโฟลเดอร์ว่าง
  2. รันคำสั่งทีละบรรทัด:
git init
git add .
git commit -m "first commit"
git branch -M main
git remote add origin https://github.com/USERNAME/my-project.git
git push -u origin main
ครั้งแรกที่ push จะมีหน้าต่างให้ล็อกอิน GitHub — ล็อกอินด้วยบัญชีของคุณ ระบบจะจำไว้ ครั้งต่อไปไม่ถามอีก

กรณี B · แก้โค้ดแล้ว commit เพิ่ม (repo มีอยู่แล้ว — วนทำทุกวัน)

  1. (ถ้าทำเป็นทีม) ดึงงานล่าสุดก่อนด้วย git pull
  2. แก้ไฟล์ในเครื่องตามต้องการ
  3. ดูว่ามีไฟล์ไหนเปลี่ยนด้วย git status
  4. เพิ่มไฟล์ + บันทึก + ส่งขึ้น:
git add .
git commit -m "บอกว่าทำอะไร เช่น เพิ่มหน้า About"
git push
รอบนี้ใช้แค่ git push เฉย ๆ (ไม่ต้องมี -u origin main แล้ว เพราะตั้งไว้ตั้งแต่ครั้งแรก) และ ไม่ต้อง git init ซ้ำ เพราะโฟลเดอร์เป็น repo อยู่แล้ว

กรณี C · แก้และ commit บนเว็บ GitHub โดยตรง (ไม่ใช้คำสั่งเลย)

เหมาะกับการแก้เล็กน้อย หรือคนที่ยังไม่ถนัดคำสั่ง — ทำบนหน้าเว็บได้เลย

  1. เปิด repo บน github.com → กดเข้าไปที่ไฟล์ที่จะแก้
  2. กดไอคอน ดินสอ (Edit this file) มุมขวาบนของไฟล์
  3. แก้ข้อความ/โค้ดในหน้าเว็บได้เลย
  4. เลื่อนลงล่างไปที่ช่อง Commit changes → ใส่ข้อความอธิบายว่าทำอะไร
  5. เลือก Commit directly to the main branch → กด Commit changes
วิธีนี้ commit ขึ้น GitHub ทันที ไม่ต้อง push (เพราะทำอยู่บน GitHub อยู่แล้ว) — แต่ถ้าคุณมีไฟล์ชุดเดียวกันในเครื่องด้วย อย่าลืม git pull ลงมาให้ตรงกันภายหลัง

นอกจากแก้ไฟล์เดิม ยังกด Add file เพื่อ Create new file (สร้างไฟล์ใหม่) หรือ Upload files (ลากไฟล์ขึ้น) ได้ด้วย — ทุกอันจบด้วยการ Commit เหมือนกัน

ข้อผิดพลาดที่เจอบ่อย + วิธีแก้

ข้อความ errorแปลว่าอะไร / วิธีแก้
remote origin already existsตั้ง origin ไว้แล้ว — ใช้ git remote set-url origin <URL> แทนการ add
repository not foundยังไม่ได้สร้าง repo บนเว็บ หรือชื่อ/URL ผิด (เช่นยังเป็นคำว่า REPO) — สร้างก่อน/ใส่ชื่อจริง
nothing to commitยังไม่มีไฟล์ หรือยังไม่ได้ git add — ใส่ไฟล์แล้ว add ก่อน
ถามล็อกอิน / 403ต้องยืนยันตัวตน — ล็อกอินด้วยบัญชีที่มีสิทธิ์ใน repo นั้น

15 ทดลองทำงานเป็นทีม (Branch + Pull Request)

สถานการณ์: ทีม 2 คน (เช่น สมชาย กับ สมหญิง) ช่วยกันพัฒนาโปรเจกต์เดียวกัน มาดูว่าต้องทำอย่างไร และบน GitHub จะแสดงอย่างไร

ขั้นที่ 1 · เจ้าของ repo เชิญเพื่อนเข้าทีม

  1. เจ้าของ repo ไปที่ Settings → Collaborators → Add people
  2. พิมพ์ username ของเพื่อน แล้วส่งคำเชิญ
  3. เพื่อนกดรับคำเชิญ (ในอีเมล หรือที่ github.com/notifications)

ขั้นที่ 2 · แต่ละคนทำงานบน branch ของตัวเอง

เพื่อจะได้ไม่แก้ทับงานกัน แต่ละคนสร้าง "กิ่งงาน" (branch) ของตัวเองแยกจาก main

  1. ที่หน้า repo กดช่องเลือก branch → พิมพ์ชื่อใหม่ เช่น feature-goodbye → Create branch
  2. แก้โค้ดและ commit ใน branch นั้น (ยังไม่กระทบ main)

สมหญิงแก้งานใน branch แยก แล้วค่อยรวม (merge) กลับเข้า main จุดวงใหญ่ = merge commit

ขั้นที่ 3 · เปิด Pull Request (ขอรวมงาน)

  1. หลัง push branch ขึ้นไป GitHub จะขึ้นปุ่ม Compare & pull request → กด
  2. เขียนหัวข้อและอธิบายว่าทำอะไร แล้วกด Create pull request

ขั้นที่ 4 · ทีมรีวิว แล้ว merge

หน้า Pull Request ที่เกิดขึ้นจะหน้าตาประมาณนี้ — เพื่อนในทีมเปิดดู รีวิว แล้วกดรวมงาน

Open เพิ่มฟังก์ชันบอกลา (goodbye)
somying ต้องการรวม feature-goodbye เข้าสู่ main
hello.py+3 -0

+3 = เพิ่ม 3 บรรทัด  ·  −0 = ลบ 0 บรรทัด (จำนวนบรรทัดที่เปลี่ยนในไฟล์)

  1. เพื่อน (หรือเจ้าของ) เปิดดู PR ที่แท็บ Files changed เห็น diff สีเขียว/แดง
  2. คอมเมนต์ขอแก้ หรือกด Approve ถ้าโอเค
  3. กด Merge pull request → โค้ดของสมหญิงรวมเข้า main

ขั้นที่ 5 · ทุกคนดึงงานล่าสุด

หลัง merge แล้ว ทุกคนในทีมควรดึง main ล่าสุดลงเครื่อง เพื่อให้ตรงกัน

git checkout main
git pull

บน GitHub จะแสดงอย่างไร

  • แท็บ Pull requests โชว์รายการ PR ทั้งหมด พร้อมสถานะ Open / Merged / Closed
  • ในแต่ละ PR มี Conversation (พูดคุย), Commits, และ Files changed (เห็น diff)
  • เมื่อรวมงานแล้ว สถานะจะเปลี่ยนเป็น Merged (ป้ายสีม่วง)
  • ประวัติ commit และ contribution graph จะบอกได้ว่าใครเพิ่ม/แก้อะไรบ้าง
อีกแบบ — Fork + Pull Request: ถ้าเป็น "คนนอกทีม" (เช่นอยากช่วยพัฒนาโปรเจกต์โอเพนซอร์สของคนอื่น) จะใช้วิธี Fork repo มาไว้ในบัญชีตัวเองก่อน แก้ไขเสร็จแล้วค่อยเปิด Pull Request กลับไปยัง repo ต้นทาง — แนวคิดรีวิว/merge เหมือนกัน
ใบทดลองทีม (จับคู่ 2 คน):
  • คน A สร้าง repo แล้วเพิ่มคน B เป็น Collaborator
  • คน B สร้าง branch ใหม่ แก้ไฟล์ แล้วเปิด Pull Request
  • คน A รีวิว PR แล้วกด Merge
  • สลับบทบาทแล้วทำอีกรอบ — ส่ง ลิงก์ PR ที่ merge แล้ว เป็นหลักฐาน

17 โบนัส: เผยแพร่เว็บนี้ด้วย GitHub Pages

เว็บที่อาจารย์กำลังอ่านอยู่นี้ สามารถอัปขึ้น GitHub แล้วเปิดเป็นเว็บจริงได้ฟรี ทำตามนี้:

  1. สร้าง repository ใหม่ (เช่น github-101) แล้วอัปโหลดไฟล์ index.html, style.css, script.js
  2. เข้าไปที่ Settings → Pages
  3. ในหัวข้อ Source เลือก branch main และโฟลเดอร์ / (root) แล้วกด Save
  4. รอสักครู่ เว็บจะออนไลน์ที่ https://USERNAME.github.io/github-101
นำลิงก์นี้ไปแชร์ให้นักศึกษาได้เลย และนักศึกษาก็ทำเว็บของตัวเองด้วยวิธีเดียวกันได้

สำคัญ: ถ้าไม่ได้เปิด Pages จะเกิดอะไร?

การอัปไฟล์ขึ้น repo เฉย ๆ ยังไม่ทำให้เป็นเว็บที่เปิดได้ — GitHub มองว่ามันเป็นแค่ "ไฟล์/โค้ด" จนกว่าเราจะเปิดสวิตช์ Pages

ไม่ได้เปิด Pages

เป็นแค่ที่เก็บโค้ด

  • เปิดดูบน github.com เห็นแค่ โค้ดต้นฉบับ ของ index.html
  • ไม่มี URL เว็บให้เข้า หน้าเว็บยังไม่รัน
  • repo ทำหน้าที่เก็บไฟล์อย่างเดียว

เปิด Pages แล้ว

กลายเป็นเว็บจริง

  • GitHub เสิร์ฟไฟล์เป็นเว็บที่ username.github.io/...
  • ใช้ index.html เป็นหน้าแรกอัตโนมัติ (ไม่ต้องตั้งค่าเพิ่ม)
  • เบราว์เซอร์เรนเดอร์ HTML/CSS/JS ออกมาเป็นหน้าเว็บ
ข้อจำกัด: GitHub Pages เปิดได้เฉพาะ เว็บแบบ static (HTML / CSS / JavaScript เท่านั้น) — ถ้าเป็นโปรแกรมฝั่งเซิร์ฟเวอร์อย่าง C หรือ Python นั้น Pages รันให้ไม่ได้ เพราะมันแค่ "ส่งไฟล์ให้เบราว์เซอร์" ไม่ได้คอมไพล์หรือรันโค้ด

ลองเอง: แก้ไฟล์ตั้งต้นแล้วอัปขึ้น GitHub ของคุณ

ผมเตรียมไฟล์ index.html ตั้งต้นไว้ให้ — มีจุดที่ให้แก้กำกับด้วยคอมเมนต์ <!-- แก้ตรงนี้ --> ลองเปลี่ยนเป็นข้อมูลของคุณ แล้วอัปขึ้น GitHub

ดาวน์โหลดไฟล์ index.html ตั้งต้น   เปิดดูตัวอย่างไฟล์

ส่วนที่ให้แก้ (ตัวอย่าง):

<!-- แก้ชื่อของคุณตรงนี้ -->
<h1>สวัสดี ฉันชื่อ ___________</h1>

<!-- แก้สาขา/ชั้นปี ตรงนี้ -->
<p class="role">นักศึกษาวิศวกรรมคอมพิวเตอร์</p>

<!-- เขียนแนะนำตัวสั้น ๆ ตรงนี้ -->
<p>นี่คือเว็บแรกของฉันบน GitHub Pages!</p>
  1. เปิดไฟล์ index.html ด้วยโปรแกรมแก้ข้อความ (เช่น VS Code หรือ Notepad) แล้วแก้ข้อความในจุดที่กำกับไว้
  2. สร้าง repository ใหม่ชื่อ my-first-web แบบ Public
  3. กด Add file → Upload files แล้วลากไฟล์ index.html ที่แก้แล้วขึ้นไป → Commit changes
  4. เข้า Settings → Pages → เลือก branch main + / (root) → Save
  5. รอสักครู่ เปิดดูเว็บของคุณที่ https://USERNAME.github.io/my-first-web

18 นักศึกษาได้อะไรบ้าง + แผนฟรี / เสียเงิน

GitHub ใช้ ฟรี ได้ตั้งแต่เริ่มต้น และนักศึกษายังได้สิทธิ์พิเศษเพิ่มอีกมาก มาดูกันว่าแต่ละแบบได้อะไร

ฟรี (Free)

ใช้ได้เลยไม่เสียเงิน

  • สร้าง repo ได้ไม่จำกัด ทั้งแบบ Public และ Private
  • เพิ่มผู้ร่วมงาน (collaborators) ได้
  • GitHub Pages — โฮสต์เว็บฟรี
  • Issues, Projects, Wiki, Discussions
  • GitHub Actions / Packages (มีโควตาฟรีต่อเดือน)

เหมาะกับนักศึกษาและงานทั่วไป — ส่วนใหญ่พอใช้แล้ว

เสียเงิน (Pro / Team / Enterprise)

จ่ายเพิ่มเพื่อใช้งานหนักขึ้น

  • Pro (รายบุคคล): โควตา Actions/พื้นที่มากขึ้น + เครื่องมือวิเคราะห์ขั้นสูง
  • Team (ทีม): จัดการสมาชิก กำหนดสิทธิ์ รีวิวแบบบังคับ
  • Enterprise (องค์กรใหญ่): ความปลอดภัยขั้นสูง (SSO), ควบคุมการเข้าถึง, ซัพพอร์ต

ใช้เมื่อทำงานองค์กร/ทีมใหญ่ หรือต้องการความปลอดภัยและโควตาสูง

นักศึกษา (Student Pack)

ของฟรีพิเศษสำหรับนักศึกษา

  • ได้ GitHub Pro ฟรี ตลอดช่วงที่เป็นนักศึกษา
  • GitHub Copilot (ผู้ช่วย AI เขียนโค้ด) ฟรี
  • เครดิตคลาวด์ (เช่น Azure, DigitalOcean) และโดเมนเว็บฟรี
  • เครื่องมือสำหรับนักพัฒนาอีกหลายสิบตัว

สมัครที่ education.github.com ด้วยอีเมลมหาวิทยาลัย + หลักฐานสถานะนักศึกษา

รายละเอียดโควตาและราคาปรับเปลี่ยนได้ตามนโยบายของ GitHub — เช็กล่าสุดที่ github.com/pricing และ education.github.com

19 แบบทดสอบความเข้าใจ

ลองทำดูเพื่อเช็กความเข้าใจ — เลือกคำตอบแล้วระบบจะบอกผลทันที

20 ใบงาน / Lab สัปดาห์นี้

ให้นักศึกษาทำตามขั้นตอนต่อไปนี้ และส่งลิงก์ repository ของตนเอง

  1. สมัครบัญชี GitHub และตั้ง username ให้เหมาะสม
  2. สร้าง repository ใหม่ชื่อ special-topic-1 แบบ Public
  3. เพิ่มไฟล์ README.md เขียนแนะนำตัวเอง (ชื่อ รหัส สิ่งที่สนใจ)
  4. อัปโหลดไฟล์โปรเจกต์เล็ก ๆ 1 ไฟล์ (เช่นเว็บหน้าแรก หรือโค้ดที่เคยเขียน)
  5. commit อย่างน้อย 3 ครั้ง พร้อมข้อความที่สื่อความหมาย
  6. (ท้าทาย) เปิด GitHub Pages ให้เว็บออนไลน์ แล้วส่งลิงก์
เกณฑ์การส่ง: ส่งลิงก์ repository มาในระบบของรายวิชา โดยใน repo ต้องมี README ที่สมบูรณ์และมีประวัติ commit ครบตามกำหนด
