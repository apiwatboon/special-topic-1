Special Topic 1 · สัปดาห์ที่ 3

Vibe Coding ทั้งโปรเจกต์
จากเว็บง่าย สู่เว็บซับซ้อน

สองสัปดาห์ที่แล้วเราสั่ง AI ทีละชิ้นเล็ก ๆ — สัปดาห์นี้ยกระดับ: สร้างเว็บไซต์ทั้งโครงการด้วย Vibe Coding ผ่าน 2 เคสจริง ตั้งแต่เว็บหน้าเดียวจบใน 4 คำสั่ง ไปจนถึงเว็บแอปที่ต้องวางแผนเป็น Milestone

เริ่มเรียน → ข้ามไปดูเคสตัวอย่าง
2เคสตัวอย่าง ง่าย/ซับซ้อน
4Milestone สำหรับงานใหญ่
5ขั้น สูตรคุมโปรเจกต์
1เว็บจริงของคุณบน Pages

เว็บ "ง่าย" กับ "ซับซ้อน" ต่างกันตรงไหน

คำว่า "ทำเว็บ" กินความกว้างมาก — ป้ายร้านออนไลน์หน้าเดียว กับระบบสั่งซื้อทั้งร้าน ใช้วิธีสั่ง AI คนละแบบกันเลย ตัวแบ่งที่แท้จริงไม่ใช่ "ความสวย" แต่คือ 3 อย่างนี้

จำนวนชิ้นส่วน

หน้าเดียว vs หลายหน้า หลายไฟล์ — ยิ่งชิ้นส่วนเยอะ ยิ่งต้องคุมให้ "เข้ากันได้" ทั้งหมด

ข้อมูลและสถานะ

เว็บที่ "จำอะไรได้" (ตะกร้า รายการโปรด บัญชีผู้ใช้) ยากกว่าเว็บที่แค่แสดงผลหลายเท่า

เส้นทางการใช้งาน

ผู้ใช้เดินทางข้ามหน้า (เลือกของ → ตะกร้า → จ่ายเงิน) จุดต่อระหว่างหน้าคือแหล่งบั๊กชั้นดี

 เว็บแบบง่ายเว็บแบบซับซ้อน
ตัวอย่างเว็บโปรไฟล์, ป้ายร้าน, หน้าแนะนำสินค้า, การ์ดเชิญงานระบบสั่งซื้อ, เว็บจองคิว, กระดานงานกลุ่ม, เว็บมีสมาชิก
โครงสร้าง1 หน้า 1 ไฟล์ จบได้หลายหน้า หลายไฟล์ ต้องออกแบบก่อน
ข้อมูลเขียนตายตัวในหน้าเว็บเปลี่ยนตามผู้ใช้ ต้องเก็บ/จำ (state)
วิธีสั่ง AIสั่งต่อเนื่องทีละส่วน 3–6 คำสั่งจบวางแผนก่อน แล้วเดินเป็น Milestone
จุดตายแทบไม่มี — พังก็สั่งใหม่บั๊กข้ามหน้า, AI ลืมบริบท, แก้จุดหนึ่งพังอีกจุด

เครื่องวัด: ไอเดียเว็บของคุณอยู่ระดับไหน?

นึกถึงเว็บที่อยากทำ แล้วติ๊กทุกข้อที่ตรง — เกจจะบอกว่าควรใช้แนวทางเคสที่ 1 หรือเคสที่ 2

ระดับความซับซ้อน 0 คะแนน
ยังไม่ได้ติ๊กอะไรเลย — เว็บแสดงผลอย่างเดียว = เคสง่ายแน่นอน

กฎทองของงานทั้งโปรเจกต์: อย่าสั่งทีเดียวทั้งเว็บ

มือใหม่เกือบทุกคนเริ่มด้วยคำสั่งเดียวยาว ๆ ว่า "ทำเว็บ...ให้หน่อย มีทุกอย่างเลย" — ผลคือได้ก้อนโค้ดยักษ์ที่ผิดครึ่งถูกครึ่ง แล้วแก้ไม่ถูกว่าจะเริ่มตรงไหน ลองเทียบสองวิธี

สั่งทีเดียวทั้งเว็บ
คุณพิมพ์:
"ทำเว็บร้านกาแฟ มีเมนู รีวิว แผนที่ ตะกร้า ระบบสมาชิก แต้มสะสม และ..."
  • ได้โค้ดก้อนยักษ์ ตรวจไม่ไหว
  • ผิดหลายจุดพร้อมกัน ไม่รู้จะแก้อันไหนก่อน
  • AI เดารายละเอียดเองเกือบหมด ไม่ตรงใจ
  • สั่งแก้จุดเดียว พังอีกสามจุด
แบ่งเป็นเฟส
คุณพิมพ์:
"เริ่มจากส่วนหัวก่อน..." → ตรวจ → "ต่อด้วยเมนู..." → ตรวจ → ...
  • โค้ดโตทีละนิด ตรวจทันทุกก้าว
  • พังตรงไหน รู้ทันทีว่ามาจากคำสั่งล่าสุด
  • ทุกส่วนตรงใจ เพราะเราคุมทิศทางตลอด
  • commit ได้ทุกจุดที่ผ่าน — ย้อนกลับง่าย
สูตร 5 ขั้นของงานทั้งโปรเจกต์: 1) คุยขอบเขตกับ AI (ยังไม่เขียนโค้ด) → 2) ให้ AI เสนอโครงและลำดับงาน → 3) สร้างโครงหลักให้เปิดได้ก่อน → 4) เติมทีละส่วน ทดสอบทุกครั้ง commit เมื่อผ่าน → 5) ขัดเงาท้ายสุด — เคสง่ายใช้ขั้น 3–5 พอ เคสซับซ้อนใช้ครบทั้งห้า

เคสที่ 1: เว็บง่าย — ร้านกาแฟหน้าเดียว จบใน 4 คำสั่ง

ดูของจริงกันเลย: สร้างเว็บร้าน "บ้านกาแฟ" ตั้งแต่หน้าว่างจนเสร็จสมบูรณ์ กด "เริ่มสร้าง" แล้วดูว่าแต่ละคำสั่งต่อยอดจากของเดิมยังไง — โค้ดด้านขวาไฮไลต์เฉพาะบรรทัดที่เพิ่มใหม่ และพรีวิวคือเว็บจริงที่เลื่อนดูได้

แชตกับ AI ยังไม่เริ่ม
กด "เริ่มสร้าง" ด้านล่าง
เพื่อดูการสร้างเว็บทั้งหน้าทีละคำสั่ง
index.html
โค้ดทั้งไฟล์จะอยู่ที่นี่ · บรรทัดใหม่ไฮไลต์สีเขียว
เว็บจริง (เลื่อนดูได้)
บทเรียนของแต่ละขั้นจะสรุปให้ตรงนี้
สังเกตให้ดี: ทั้ง 4 คำสั่งไม่มีคำว่า "ทำเว็บทั้งหมดให้หน่อย" เลย — ทุกคำสั่งสั้น เฉพาะเจาะจง และอ้างถึงของที่มีอยู่แล้ว ("ใต้ส่วนหัว", "ท้ายหน้า") นี่คือหน้าตาของ Vibe Coding ระดับโปรเจกต์ในเคสง่าย

เคสที่ 2: เว็บซับซ้อน — ระบบสั่งกาแฟออนไลน์ทั้งร้าน

คราวนี้โจทย์ใหญ่ขึ้นมาก: ร้านเดิมอยากได้ระบบสั่งซื้อจริง — ลูกค้าเลือกเมนู หยิบใส่ตะกร้า ปรับจำนวน เห็นยอดรวม และร้านเห็นคำสั่งซื้อ งานแบบนี้สั่งรวดเดียวไม่มีทางรอด ต้องเดินเป็น Milestone — คลิกดูทีละด่าน แต่ละด่านมี prompt ตั้งต้นให้คัดลอกไปใช้ได้เลย

เป้าหมาย: ได้ "พิมพ์เขียว" ของทั้งระบบก่อนลงมือ — มีหน้าอะไรบ้าง เก็บข้อมูลอะไร และสร้างลำดับไหนก่อนหลัง แล้วบันทึกเป็นไฟล์ SPEC.md ไว้ใช้อ้างอิงตลอดโปรเจกต์

prompt ตั้งต้น
ผมอยากทำเว็บสั่งกาแฟออนไลน์ของร้านเดียว ผู้ใช้เลือกเมนู
หยิบใส่ตะกร้า ปรับจำนวน และเห็นสรุปยอด — อย่าเพิ่งเขียนโค้ด
ช่วยเสนอ: (1) รายการหน้าทั้งหมด (2) ข้อมูลที่ต้องเก็บ
(3) ลำดับการสร้างทีละขั้นจากง่ายไปยาก พร้อมเหตุผล

เช็กก่อนผ่านด่าน: อ่านแผนแล้วเข้าใจทุกข้อ · ตัดฟีเจอร์ที่ยังไม่จำเป็นออก (แต้มสะสม? ไว้ทีหลัง!) · บันทึกเป็น SPEC.md ในโปรเจกต์

กับดัก: ข้ามด่านนี้เพราะ "อยากเห็นเว็บเร็ว ๆ" คือสาเหตุอันดับหนึ่งของโปรเจกต์พัง — โค้ดที่ไม่มีแผนจะมั่วตั้งแต่โครง และแก้ทีหลังแพงกว่าวางแผนก่อนเสมอ

เป้าหมาย: โครงกระดูกของทั้งเว็บ — หน้าเมนู หน้าตะกร้า หน้าสรุปออเดอร์ เปิดได้ครบ มีแถบนำทางเชื่อมถึงกันหมด เนื้อหาใช้ข้อมูลตัวอย่างไปก่อน ยังไม่ต้องมีฟีเจอร์จริง

prompt ตั้งต้น
นี่คือ SPEC ของโปรเจกต์: [วางเนื้อหา SPEC.md]
ช่วยสร้าง "โครง" ก่อน: ทุกหน้าตาม SPEC เปิดได้จริง
มีแถบเมนูนำทางครบ ใช้ข้อมูลตัวอย่างไปก่อน
ยังไม่ต้องทำระบบตะกร้า — เอาแค่โครงสวย ๆ ที่คลิกได้ทั่ว

เช็กก่อนผ่านด่าน: คลิกไปได้ครบทุกหน้า ไม่มีลิงก์ตาย · หน้าตาโดยรวมโอเค · git commit "โครงเว็บครบทุกหน้า"

กับดัก: อย่าใจร้อนใส่ฟีเจอร์จริงในด่านนี้ — ถ้าโครงยังขยับอยู่แล้วรีบใส่ระบบตะกร้า พอแก้โครงทีเดียว ฟีเจอร์พังหมดต้องทำใหม่

เป้าหมาย: เส้นทางหลักของผู้ใช้ทำงานได้จริงครบวงจร: กด "ใส่ตะกร้า" จากหน้าเมนู → ไปหน้าตะกร้าเห็นรายการ → เพิ่ม/ลดจำนวน ลบรายการ → ยอดรวมคำนวณถูก

prompt ตั้งต้น
โครงเว็บเสร็จแล้ว ต่อไปทำระบบตะกร้าตาม SPEC:
กดใส่ตะกร้าจากหน้าเมนูได้ หน้าตะกร้าแสดงรายการ
เพิ่ม/ลดจำนวน ลบรายการได้ และยอดรวมต้องคำนวณใหม่ทุกครั้ง
ทำทีละหน้า เริ่มจากปุ่ม "ใส่ตะกร้า" ก่อน

เช็กก่อนผ่านด่าน (ทดสอบเคสแปลก ๆ ด้วย): ใส่ของชิ้นเดิมซ้ำ ๆ · ลดจำนวนจนเหลือ 0 · ลบจนตะกร้าว่าง · กดปุ่มรัว ๆ — ยอดรวมต้องถูกทุกกรณี · commit เมื่อผ่าน

กับดัก: บั๊กของเว็บซับซ้อนชอบซ่อนที่ "ข้อมูลข้ามหน้า" (หยิบของหน้านี้ ไปโผล่อีกหน้า) เหมือนบั๊ก 120→1201 ของสัปดาห์ก่อน แต่ตามยากกว่า — ทดสอบด่านนี้ให้หนักที่สุดในโปรเจกต์

เป้าหมาย: ตะกร้าไม่หายเมื่อรีเฟรชหรือปิดเบราว์เซอร์ (ใช้ localStorage — กล่องเก็บข้อมูลเล็ก ๆ ในเบราว์เซอร์ของผู้ใช้) แล้วค่อยขัดเงา: responsive, อนิเมชัน, ความสวย สุดท้าย push ขึ้น GitHub Pages

prompt ตั้งต้น
ระบบตะกร้าทำงานถูกแล้ว ช่วยทำให้ตะกร้า "จำได้"
ด้วย localStorage: รีเฟรชหรือปิดเปิดใหม่ รายการต้องยังอยู่
เสร็จแล้วอธิบายให้ผมฟังด้วยว่าเก็บอะไรไว้ตรงไหน

เช็กก่อนผ่านด่าน: ใส่ของ → ปิดเบราว์เซอร์ → เปิดใหม่ ตะกร้ายังอยู่ · ลองบนจอมือถือ · commit แล้ว push ขึ้น GitHub Pages

กับดัก: อย่าเอา "ขัดเงา" ขึ้นก่อนฟีเจอร์เสร็จ — เว็บสวยที่คิดเงินผิด แย่กว่าเว็บเรียบ ๆ ที่คิดเงินถูก
หัวใจของเคสซับซ้อน: แต่ละ Milestone คือ "เว็บที่ยังใช้งานได้" เสมอ — เราไม่เคยอยู่ในสภาพ "พังยาวสามวันเพราะทำหลายอย่างพร้อมกัน" และทุกด่านจบด้วย commit จึงย้อนกลับได้ตลอดเวลา

เทคนิคคุมโปรเจกต์ยาว ๆ ไม่ให้หลุดมือ

พอโปรเจกต์ใหญ่ขึ้น ศัตรูตัวใหม่จะโผล่มา: AI จำรายละเอียดทั้งหมดไม่ไหว (บทสนทนายาวเกิน มันจะเริ่ม "ลืม" ของที่คุยไว้ตอนต้น) — มืออาชีพจัดการแบบนี้

  1. SPEC.md คือสมองสำรอง — เก็บแผนไว้ในไฟล์ ไม่ใช่ในแชต เริ่มเซสชันใหม่เมื่อไหร่ก็วาง SPEC ให้ AI อ่านก่อน มันจะรู้บริบททั้งหมดใน 10 วินาที
  2. ขอ "สรุปสถานะ" เป็นระยะ — พิมพ์ว่า "สรุปให้หน่อยว่าตอนนี้ทำอะไรเสร็จแล้ว เหลืออะไร" แล้วเก็บคำตอบต่อท้าย SPEC — มีประโยชน์มากเวลาปิดคอมแล้วกลับมาทำต่อ
  3. หนึ่งเซสชันแชต ต่อหนึ่งภารกิจ — แชตที่คุยยาวข้ามหลายเรื่องจะยิ่งมั่ว เปลี่ยน Milestone เมื่อไหร่ เปิดแชตใหม่พร้อมวาง SPEC ดีกว่า
  4. commit คือเซฟพอยต์ของเกม — ผ่านด่านไหน เซฟด่านนั้น (ทวน สัปดาห์ 1) ถ้า AI แก้แล้วพังหนัก ก็ย้อนกลับไปเซฟล่าสุดได้ใน 1 คำสั่ง
  5. ให้ AI ทดสอบตัวเอง — ก่อนปิดงานทุกครั้ง ลองสั่งว่า "ช่วยไล่ดูโค้ดทั้งหมด มีจุดไหนน่าจะเป็นบั๊กหรือช่องโหว่ไหม" — จับได้เยอะกว่าที่คิด

เครื่องมือก็มีผล: แชตธรรมดา vs เอเจนต์

เคสง่ายใช้แชตธรรมดา (Claude, ChatGPT บนเว็บ) สบาย ๆ แต่เคสซับซ้อนที่มีหลายไฟล์ เครื่องมือแบบ "เอเจนต์" (เช่น Claude Code) จะได้เปรียบชัดเจน

 แชตธรรมดาเอเจนต์ (เช่น Claude Code)
เหมาะกับเว็บ 1–2 ไฟล์ งานชิ้นเล็กโปรเจกต์หลายไฟล์ งานทั้งระบบ
การแก้ไฟล์เราคัดลอกโค้ดไปวางเองแก้ไฟล์ในเครื่องให้เองโดยตรง
บริบทโปรเจกต์ต้องวางโค้ดให้ดูใหม่ทุกครั้งเปิดอ่านไฟล์ในโปรเจกต์ได้เอง
การทดสอบเรารันเองแล้วรายงานผลกลับรันคำสั่ง/ทดสอบแล้วแก้ต่อเองได้
สิ่งที่เราต้องทำเป็นคนกลางส่งของไปมาตั้งโจทย์ กำกับ และตรวจรับ
พลังมากขึ้น = ต้องตรวจมากขึ้น: เอเจนต์แก้หลายไฟล์ได้ในคำสั่งเดียว แปลว่าพังหลายไฟล์ได้ในคำสั่งเดียวเช่นกัน — กฎ "อย่ารันสิ่งที่ไม่เข้าใจ" และ "commit ก่อนให้แก้ครั้งใหญ่" ยิ่งสำคัญเป็นสองเท่า

โจทย์ใหญ่: สร้างเว็บทั้งโปรเจกต์ของคุณเอง

โจทย์มี 2 แทร็กให้เลือกตามความมั่นใจ — แทร็ก A เว็บหน้าเดียวแบบเคสที่ 1 หรือ แทร็ก B มินิเว็บแอปมีระบบจำข้อมูลแบบเคสที่ 2 ติ๊กเช็กพอยต์ได้ทีละข้อ (บันทึกให้อัตโนมัติในเครื่อง)

ภารกิจ: เว็บ 1 หน้าธีมอิสระ (ร้านสมมุติ ชมรม วงดนตรี ทีมอีสปอร์ต — ห้ามซ้ำโปรไฟล์สัปดาห์ที่แล้ว) อย่างน้อย 4 ส่วน: ส่วนหัว เนื้อหา 2 ส่วน และส่วนติดต่อ
